1. MongoDB es case-sensitive siempre por defecto, sin embargo, se debe poner de nuevo para normalizar el valor antes de persistirlo. Debido a que, si no se agrega el .toLowerCase(), al buscar algún dato como "McQueen" o "mcqueen", se almacenaría solo uno de ellos, como elementos separados, no obstante, al usar el .toLowerCase() se guardan ambos en la DB.

2. Ambas validaciones operan en capas distintas con distintas responsabilidades, en el isValidObjectId(), el error se lanza en el servicio, manualmente con un BadRequestException. Por otro lado, el ParseMongoIdPipe atrapa el error en el controller y se lanza antes de que entre, de tal manera que el servicio NestJS lo convierte en 400 Bad Request.

3. Solo se usa el try/catch en create debido a que findAll hace una lectura tipo find() y las lecturas nunca violan restricciones de base de datos, ya que no hay unividad, no hay índices ni algo que MongoDB pueda rechazar en un find. Por otro lado, create escribe un documento nuevo, donde sú puede haber algún error de índice único, en caso de que ya exista un documento con el mismo nombre. De tal manera, si se llegara a eliminar el try/catch del create, MongoDB lanzaría error 11000:
NestJS no sabría qué hacer con ese error de Mongoose — no es una HttpException. Lo trata como excepción no controlada y devuelve 500 Internal Server Error.

4. El método hace dos queries: this.fundOne(id), que es un findById para verificar que existe el elemento, y car.updateOne(updateCarDto) que hace el update en MongoDB Puede haber una diferencia entre lo que retorna la API y lo que está en MongoDB, debido a que el return es { ...car.toJSON(), ...updateCarDto } que es una fusión en memoria. Pero updateCarDto viene del cliente tal cual, sin pasar por transformaciones de Mongoose.

5. Con forRoot(process.env.MONGODB_URL || '...'), JavaScript evalúa process.env.MONGODB_URL en el momento en que se ejecuta ese módulo, es decir, cuando Node.js importa app.module.ts. Si en ese momento, las variables de entorno aún no han sido cargadas, process.env.MONGODB_URL es undefined y se usa el fallback hardcodeado. En cambio, forRootAsync con useFactory recibe la factory como una función que NestJS invoca más tarde, después de que el sistema de intección de dependencias ya resolivó todo.

Algunos errores de imports en módulos son olvidar importar el CarsModule en AppModule, el cual aparece al arrancar la aplicación, no en el primer request NestJS construye el grafo de dependencias al inicializar, detecta que CarsController o CarsService no están registrados en ningún módulo alcanzable desde AppModule, y lanza un error de tipo Nest can't resolve dependencies o simplemente no registra las rutas.
   También el importar CarsModule pero olvidar MongooseModule.forFeature dentro del mismo. Así, NestJS sí levanta el módulo pero al momento de intentar inyectar @InjectModel(Car.name) en el CarsService no encuentra el token del modelo en el contenedor y lanza un error del tipo 'Nest can't resolve dependencies of the CarsService (?). Please make sure that the argument CarModel at index [0] is available in the CarsModule context.' Por loq ue en el archivo de module.ts falta el MongooseModule.forFeature { name: Car.name, schema: CarSchema }  en sus imports.

7. Esto se puede usar, debemos solo hacer uso de un query en lugar de dos, ya que llamar findOne primero para luego borrar es un patrón del tipo "read-before-write" por lo que se gastarían innecesariamente espacios en la DB. Con deleteOne se hace todo en una sola operación.
   deletedCount puede ser 0 como objectId válido cuando el documento antes existía pero fue eliminado entre el momento en que el cliente obtuvo el ID y el momento en que ejecuta el Delete.

8. La ventaja arquitectónica es que los pipes en NestJS son parte del request lidecycle y operan en la capa de transporte HTTP, los cuales deben transdormar y validar la entrada antes de que llegue a la lógica del negocio. Si la validación está en el servidio, el servicio asume una responsabilidad que no le corresponde, lo que hace más difícil reutilizar la misma. Si se quitara el Injectable, NestJS instancia el pipe directamente como una clase porque se está pasando la clase como referencia, no como una instancia. En ese caso, si el Pipe no tiene dependencias inyectadas, funcoina igualm pero el injectable es necesario solo si el pipe necesita que NestJS intecte algo en su constructor.

9. En el escenario de useGlobalPipes no cambia el comportamiento, ya que son configuraciones que se aplican antes del app.listen(), NestJS las consolida al momento de levantar el servidor, además, el orden entre ellas no importa porque ninguna depende de las otras.
   En el escenario de enableColors() después de app.listen() sí hay un problema, ya que app.listen() es una llamada asíncrona que empieza a aceptar conexiones que, en el caso de que se registrara después, existe una ventana de tiempo en la que el servidor ya acepta requests pero el middleware aún no está activo.
